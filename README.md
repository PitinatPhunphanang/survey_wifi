# Wi-Fi Survey Pro 📶

โปรเจกต์สำหรับทดสอบและเก็บข้อมูลประสิทธิภาพของเครือข่าย Wi-Fi (Site Survey) แบบครบวงจร ตั้งแต่การเก็บข้อมูลภาคสนามด้วยโปรแกรม Python ไปจนถึงการแสดงผลรายงานด้วย Next.js Dashboard แบบเรียลไทม์

---

## 🛠 การอัปเดตระบบล่าสุด (Changelog)

ระบบได้รับการยกระดับจากการใช้ไฟล์ Excel เพียงอย่างเดียว มาเป็นระบบฐานข้อมูลเต็มรูปแบบ เพื่อให้ข้อมูลเชื่อมโยงกันแบบอัตโนมัติ (Automated Data Pipeline):

1. **เพิ่ม Database:** ติดตั้ง PostgreSQL ผ่าน Docker Compose เพื่อเป็นศูนย์กลางในการเก็บข้อมูล
2. **ปรับปรุงโปรแกรมฝั่ง Python (`Wifi.py`):** ให้ส่งข้อมูลผลเทสตรงเข้า Database ทันทีที่เทสเสร็จ (พร้อมกับสำรองข้อมูลลง Excel แบบเดิม)
3. **ยกเครื่องหน้า Dashboard (Next.js):**
   - ถอดระบบอัปโหลดไฟล์ Excel แบบ Manual ออก
   - เพิ่มระบบ API (`/api/survey` และ `/api/trace`) ให้เว็บไปดึงข้อมูลจาก Database สดๆ
   - เพิ่มเมนูด้านซ้าย (Sidebar) สำหรับเลือกตึก (Building) และชั้น (Floor) ที่ต้องการดู
   - แสดงผลกราฟวิเคราะห์แต่ละชั้น (Floor Report) และหน้าข้อมูลการเชื่อมต่อเส้นทาง (Traceroute Analysis) ไว้ในหน้าเดียว
   - เปลี่ยนเมนูและส่วนติดต่อผู้ใช้งานเป็น **ภาษาไทย** เพื่อง่ายต่อการใช้งาน (แต่คงศัพท์เทคนิคไว้)

---

## 💾 หลักการจัดเก็บข้อมูลเข้า Database (Data Ingestion)

การนำข้อมูลเข้าสู่ฐานข้อมูลจะถูกจัดการโดยสคริปต์ **Python (`Wifi.py`)** หลังจากที่โปรแกรมทำงานเทส (Ping, iPerf, Traceroute) เสร็จสิ้นในแต่ละจุดทดสอบ:

1. **การเชื่อมต่อ:** ใช้ไลบรารี `psycopg2` ในการเชื่อมต่อไปยัง PostgreSQL ที่รันอยู่บน Docker
2. **การบันทึกผลเทสหลัก (`survey_results`):** 
   - โปรแกรมจะรวบรวมข้อมูลทั้งหมด (เช่น RSSI, Ping, Download/Upload Mbps, Jitter, Packet Loss) และใช้คำสั่ง `INSERT INTO survey_results`
   - ในขั้นตอนนี้จะใช้คำสั่ง `RETURNING id` เพื่อดึงรหัสของผลเทสปัจจุบัน (Survey ID) กลับมา
3. **การบันทึก Traceroute (`trace_hops`):**
   - โปรแกรมจะนำ `id` ที่ได้จากข้อ 2 ไปผูกกับข้อมูล Hop แต่ละจุดของ Traceroute
   - ทำการ `INSERT INTO trace_hops (survey_id, hop, ip, ...)` เพื่อบันทึกข้อมูลย่อยทีละบรรทัด
4. **ความปลอดภัย:** หากเกิดปัญหา (เช่น เปิด Database ไม่ติด) ระบบจะยังคงเซฟข้อมูลลงไฟล์ Excel ให้ตามปกติเพื่อป้องกันข้อมูลสูญหาย

## 📊 หลักการดึงข้อมูลไปแสดงผล (Data Retrieval & Visualization)

หน้าเว็บ Dashboard (Next.js) จะถูกทำหน้าที่เป็นฝั่งสำหรับดึงและแสดงข้อมูล (Read-Only) มีกลไกดังนี้:

1. **ฝั่ง Backend (API Routes):**
   - สร้างไฟล์ `/api/survey/route.ts` และ `/api/trace/route.ts` 
   - ใช้ไลบรารี `pg` เพื่อส่งคำสั่ง SQL เช่น `SELECT * FROM survey_results ORDER BY timestamp DESC` 
   - ข้อมูล Traceroute จะถูกดึงโดยใช้คำสั่ง `JOIN` ระหว่างตาราง `trace_hops` และ `survey_results` เพื่อหาข้อมูลสถานที่ (ตึก/ชั้น) ของการเทสครั้งนั้น
   - ส่งออกข้อมูลเป็นรูปแบบ JSON
2. **ฝั่ง Frontend (Dashboard UI):**
   - ทันทีที่เปิดหน้าเว็บ หรือกดปุ่ม **"ดึงข้อมูลล่าสุด"** ระบบจะวิ่งไปเรียก API เพื่อขอข้อมูล JSON 
   - ระบบจะนำข้อมูลมาประมวลผลหา **รายชื่อตึกและชั้นทั้งหมด** เพื่อสร้างแถบเมนูด้านซ้าย
   - เมื่อผู้ใช้ **คลิกเลือกชั้น** ระบบจะกรอง (Filter) เฉพาะข้อมูลของชั้นนั้น และส่งต่อให้ไลบรารี `recharts` สร้างเป็นกราฟวิเคราะห์ (KPI, RSSI, Throughput, Jitter, Packet Loss)
   - ข้อมูล Traceroute ก็จะถูกจับคู่และแสดงผลบนตารางและ Timeline ทันที

## 🚀 วิธีการรันโปรเจกต์ (How to run)

1. **รัน Database:**
   ```bash
   เปิดdocker-compose up -d
   ปิดdocker compose down
   ```เระ
2. **รัน Web Dashboard:**
   ```bash
   npm install   # (ทำเฉพาะครั้งแรก)
   npm run dev
   ```
   เปิดเว็บได้ที่ `http://localhost:3000`
3. **รันตัวเทส Wi-Fi (ฝั่งผู้ปฏิบัติงาน):**
   ```bash
   # ทำในโฟลเดอร์ที่มีไฟล์ Wifi.py
   python Wifi.py
   ```
